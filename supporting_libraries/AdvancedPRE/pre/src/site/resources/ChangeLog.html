Enhancements:

 

1.   Added a new API as requested by JBEAM that indicates whether the PRE is actively scanning or not. When a stop request is processed by PRE then PRE stops picking up new requests that were lined up and will wait for the currently running jobs to finish their work. If a running job is pro-creating newer jobs and then waits on these new jobs to finish then the execution goes into a continuous loop and will exit only when killed or when the max stuck thread limit of all running jobs is reached. Through this new API one can identify if such a situation has occurred and is expected to then quit the processing in a safe manner.

Dated: 20100827 16:44:08

 Support Available till Dec 31, 2011

 

Enhancements:

 

1.     Now compatible only with JDK 1.6 and above. PRE now uses 1.6 features.

2.     ACTIVE / PASSIVE clustering  is now made available in this version of PRE. For this to happen, PRE makes use of Hazelcast, the details can be found here http://www.hazelcast.com/.

Case 1: Say we have PRE A and PRE B participating in a cluster and PRE A is currently ACTIVE and B is in PASSIVE mode. In case PRE A gets killed or crashes then the jobs that were being executed by A will fail over to B and B will be made ACTIVE. Please note that the PRE B will re-start the jobs that were failed over. This means that if there is any need of serialization then that needs to be handled by the job itself. The default hazelcast.xml is made available in the resources folder. In case of n  PREs participating in the cluster then the PRE that gets registered first on the Hazelcast will be made active and all others will be made PASSIVE. In case the Active PRE is killed or crashes then the subsequent PRE that was registered second on the Hazelcast will be made Active and all others will maintain their current PASSIVE state.

In case of a STOP request processed by the Active PRE; then all other PASSIVE instances of PRE participating in the cluster will shut themselves down. For Hazelcast configuration there is one hazelcast.xml file that must be replicated on all participating servers by duly updating the IP address of each participating server.

Enhancements:

 

1.     The JDBC Data Source is now property driven. Introduced two new properties to indicate which data source (in turn, which pool) to be used for StandAlone Requests and Group Requests. Refer to dsforstandaloneeng and dsforgroupeng properties in pr.properties. If the property is left blank then the PRE will not start.

                       

2.     The new API has been exposed in the Email class to accept the sender email address. EMailType class has been introduced that define two types of Emails namely HTML and TEXT. This type can be set in the EMail object by calling email.setEMailType(EMailType.HTML); The default message header and/or footer will not be added by the CMailer class. If the type is  HTML then it is expected that the getMessageBody() method from Email returns an HTML. The property defaultemailoftype is added to mail.properties.

3.     Introduced a functionality for STOPing a request through the WEB interface. If the JOB implements ITerminateProcess interface then it is possible to invoke stop on this JOB while it is being executed by PRE. The STOP does not guarantee that the process will stop itself unless the control is with the JOB. If the JOB has invoked a database procedure/function and the control is not with the JOB then till the time the control returns back to the JOB the JOB may not stop itself. PRE will just invoke the terminate(String..) method on the JOB indicating that the JOB must stop its processing.

4.     Introduced a Data Source factory. Now PRE is not tied to the JDBCPool; but is abstracted. To remove the dependency, one needs to implement the interface stg.pr.engine.datasource.IDataSourceFactory. This class exposes 3 methods one to initialize the data source factory, second to shutdown the factory and third to retrieve DataSource by passing the name of the datasource. The default implementation of PRE still makes use of the JDBCPool API. The initialize method accepts File as a parameter. This file path and name is stored in the property dataSourceFactoryConfigFile in pr.properties. The class that implements IDataSourceFactory is stored in pr.properties against property dataSourceFactory. Now, one can make use of the same data source for both the stand-alone and/or group request engines.

5.     Introduced Findbugs report in the Zip file. Please note that the jar file is obfuscated using Proguard. The findbugs report is executed on the source before obfuscation. Release V1.0R24.00.P011 (Major Release)

Dated:20080711 08:55:30

Backward Compatibility Issues may be noted. Classes that reference package stg.pr.engine.beans will need modification(s) to catch SQLException thrown by these beans. Table structures modified.

 

First, the Bug Fixes

1.     Last date of the month frequency had a bug. The date parameters associated with a request that has the frequency LD_MONTH_FREQUENCY used to get advanced by a year instead of a month. This has been fixed in this release. This bug is present from V1.0R15.x.x. The JOB scheduling used to work on a monthly basis as desired.

 

2.     CExport had a bug. This class is not used in the PRE but is a utility class to export data to flat files as per the specifications stored in properties file.

 

3.     If the parameters associated with a request are not correctly associated with the data type format as defined in the pr.properties (specifically for the date type parameters) then the request does get error out but an associated schedule remains in Active status. This schedule remaining active will not cause any issue but an email notification will not be sent if it is still Active. This version fixes this issue. The schedule will be marked as Terminated and an appropriate email notification will be e-mailed provided the mail notification is on.

 

Now Enhancements

 

A) 5 New frequencies have been added.

 

FIRST_IN_YEAR_FREQUENCY

SECOND_IN_YEAR_FREQUENCY

THIRD_IN_YEAR_FREQUENCY

FOURTH_IN_YEAR_FREQUENCY

LAST_IN_YEAR_FREQUENCY

 

These are frequencies for a specific <day> in the <n>th week recurring on a Yearly basis where day represents Sunday through Saturday and n represents First, Second, Third, Fourth and Last week of the month. The start date is used for calculating the month in which it has to recur on a yearly basis.

 

B) Exception logging improved. The SQLExceptions are now thrown from the beans.

 

C) Changes made to discard the email_ids field from the process_request_schedule table. Instead the email id field from process_request is being used to send all schedule related email notification to the recipients.

 

D) Calendar WORK Day check is now added. This means that a Calendar or a pre-programmed frequency can now be associated with a Calendar that stores the WORK/Non-WORK days. If next recurrence of the schedule date is a non-work day then PRE will advance the date as per the skip flag associated with the schedule.

 

C) It is not always necessary to associate a Calendar with a schedule. If a week day check is associated with the schedule and if a schedule date happens to be a SATURDAY or SUNDAY then the schedule can be advanced to the next week day based on the skip flag.

 

A skip flag can have the values "NA", "D+", "D-". Where NA is the skipping of the schedule should not be done and date is arrived as per the frequency associated will be used as is. If the skip flag is D+ or D- then there must be either a weekday check set to Y or should have a associated Calendar. The day will be increased by 1 in case the flag is D+ and day will be advanced by 1 (-1) in case the flag is D-.

 

The schedule will be terminated if the associated skip flag is D- and the actual job execution date goes below the calendar scheduled date. PRE keeps a track of the number of these checks (weekday check and ICalendar check) are applied. The counter is increased by one when either or both of the checks are applied. If this counter goes beyond the counter set in property icalendarinfiniteloopterminationcounter in pr.properties then the schedule is terminated. The case where the skip flag is SS then the counter is increased by 1 per number of checks applied. Example a Week day check is applicable then on applying this check the counter is increased by 1. Also, if the same schedule is associated with a ICalendar then on applying the ICalendar check the counter is increased by 1. This avoids infinite looping if the ICalendar always returns false.

 

D) Column menu_id renamed to job_id (JOB_ID is a unique identifier for the JOB. This field is just a place holder.)

E) Column job_name newly added. (JOB_ID is a unique identifier for the JOB. This field is just a place holder.)

Release 23.00.001 (Major Release)

Dated:          20060701 15:10:10 EST

 Support: Not available.

 

Enhancements

API Change. Table/Structural changes.

 

A complete overhauling of the engine is done. Added a Controller class that will be responsible to start, boot, shutdown the engine. Also, the JDBCPool is being used in this and the proprietry pool has been removed. A poolconfig.xml has been introduced, which defines two pools ST and GRP. The maximum capacity of these pools can differ unlike the previous one. The maximum capacity determines the number of threads that can be spawned by the Engine. The maximum threads that can be spawned will be always one less than the

maximum capacity of the pool.

 

VERY IMPORTANT CHANGES

1.     The PRE will be bounced if all the execution threads of the Stand Alone engine are identified as STUCK and the PREis not able to service any requests further. The bouncing of the Engine will be done if all the threads that the PRE spawns crosses the STUCK THREAD MAX LIMIT identified against each thread (request). Suppose we have 3 requests and lets assume that the 3 requests has 20, 40 and 60 as stuck thread max limit and the PRE is configured for 3threads only. In such a case, the PRE will bounce if all 3 are running and they cross the 60 minute max limit marked against the 3 request even though the other 2 requests have crossed their respective stuck thread max limits.

2.     The fixed date is now added as a facility in the scheduling. E.g. In earlier versions of PRE, if we had to schedule a job on every 28th day then it was not possible. Because if 28th Feb was a last day of the month then PRE used to advance it to the last day of March. This is now taken care of. If you want to have PRE, schedule on every 28th of the month then assign fixed_date as 'Y'. If you want every 31 then the schedule will skip till it finds a month that has the date 31 in case the fixed date is set to 'Y'.

 

Changes are made in startEng.sh, stopEng.sh and bootEng.sh.

 

STRUCTURAL CHANGES:

A new column is added in the PROCESS_REQUEST table. The following queries needs to be executed in order for PRE to work.
